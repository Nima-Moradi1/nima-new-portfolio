# Internationalization

The portfolio uses `next-intl` with English, Persian, and German. English is the default locale and has unprefixed canonical URLs; Persian and German use locale prefixes.

| Locale | Direction | Home  | XO Arena case study     |
| ------ | --------- | ----- | ----------------------- |
| `en`   | LTR       | `/`   | `/projects/xo-arena`    |
| `fa`   | RTL       | `/fa` | `/fa/projects/xo-arena` |
| `de`   | LTR       | `/de` | `/de/projects/xo-arena` |

Locale detection is enabled, so the middleware may redirect a visitor to a preferred non-English locale. English URLs themselves remain unprefixed.

## Architecture

- `src/i18n/routing.ts` is the source of truth for supported locales, the default locale, URL-prefix behavior, and text direction.
- `src/proxy.ts` runs the `next-intl` middleware for application routes while excluding APIs, Next.js internals, and files with extensions.
- `src/i18n/request.ts` validates the request locale, dynamically loads its message module, and sets the application time zone.
- `src/i18n/navigation.ts` exports locale-aware `Link`, redirect, pathname, and router helpers. Use these instead of `next/link` for application navigation.
- `src/i18n/paths.ts` builds canonical and language-alternate URLs, including `x-default`.
- `src/app/[locale]/layout.tsx` validates the locale, sets `lang`, `dir`, and `data-locale`, loads messages, provides `NextIntlClientProvider`, and emits localized metadata and JSON-LD.
- `src/components/i18n/locale-switcher.tsx` changes locale without losing the current pathname or hash.

The root layout only imports global CSS. The middleware internally supplies the locale segment needed by the `[locale]` route tree.

## Message and portfolio catalogs

Each locale has three modular catalogs and one merge module:

```text
src/messages/en/common.json
src/messages/en/portfolio.json
src/messages/en/xo-arena.json
src/messages/en.ts
```

`fa` and `de` mirror this structure. The merge module spreads `common.json` at the root and exposes structured content as `Portfolio` and `XoArena`.

English is the schema source of truth. Every locale must preserve:

- all object keys and value types;
- array lengths and ordering;
- stable `id` values;
- ICU placeholders such as `{name}`, `{number}`, and `{progress}`;
- factual claims, product/technology names, URLs, email addresses, and asset paths unless a reviewed locale-specific asset replaces them.

`src/types/portfolio.ts` defines the structured portfolio contract. `materializePortfolio` in `src/content/portfolio.ts` overlays the runtime contact email without changing localized content. Server code can materialize `messages.Portfolio`; client components use `usePortfolio()`. The exported English `portfolio` remains a compatibility default for non-localized code and tests.

Use stable IDs as React keys. Do not derive structure from translated punctuation or translated labels.

## RTL, fonts, and bidirectional content

Persian uses `dir="rtl"` and Vazirmatn; English and German use Geist, with Geist Mono retained for code and technical labels. The locale layout loads all three font variables. The generated Open Graph image also embeds the local Vazirmatn font so Persian text renders reliably.

Global styling favors logical properties such as `margin-inline`, `padding-inline`, `inset-inline-*`, and `border-inline-*`. Direction variables in `globals.css` mirror directional decoration and icons. Persian-specific rules remove Latin uppercase/letter-spacing treatments and adjust line height and wrapping.

Mixed-direction values require isolation:

- wrap technology names, code, email addresses, URLs, and technical numbers in `<bdi>` or an existing bidi helper;
- keep email/URL inputs `dir="ltr"`; free-text fields use `dir="auto"`;
- do not visually mirror screenshots, videos, rﾃｩsumﾃｩ previews, or code canvases.

Direction also affects behavior. The experience readers reverse horizontal keys and arrows for RTL, and the Three.js book reverses its page turn. The capability timeline uses `src/lib/logical-scroll.ts`, which normalizes the browser-specific RTL `scrollLeft` models. Do not replace it with direct `scrollLeft` arithmetic.

## Metadata and install surfaces

The locale layout and XO Arena page generate localized titles, descriptions, keywords, canonical URLs, `hreflang` alternates, Open Graph locales, Twitter data, and image alt text. When changing routes, update the route list in `src/app/sitemap.ts`; it emits every locale plus language alternates.

Manifests are generated by `src/lib/create-manifest.ts`:

- `/manifest.webmanifest` serves English;
- `/{locale}/manifest.webmanifest` serves a prefixed locale;
- name, description, start URL, language, and direction are locale-aware;
- the stable `/` app ID and scope keep every locale inside one installable app.

The Open Graph locale maps currently live in both `src/app/[locale]/layout.tsx` and `src/app/[locale]/projects/xo-arena/page.tsx`. Keep both maps synchronized when adding a locale.

## Contact localization

The client builds its Zod schema from `ContactForm.validation`, renders localized labels and states, and sends `x-portfolio-locale` to `/api/contact`. The API validates that header, falls back to English when invalid, creates the same localized schema, and translates response messages, mailto subjects, provider subjects, and email bodies from `ApiContact`. Stable response `code` values are intentionally not translated.

When editing form constraints, update the client/server schema and every locale's validation copy together.

## Embedded-language media policy

Project screenshots, the XO Arena walkthrough, and the rﾃｩsumﾃｩ are authentic captures. Some contain English or Persian UI. They remain unmirrored and unaltered across locales unless a genuine localized capture or document is available. Localize the surrounding alt text, captions, and explanation; never generate or paint translated text over a product capture in a way that misrepresents the shipped product.

Current media ownership is split as follows:

- rﾃｩsumﾃｩ PDF/preview and project-card image paths live in each locale's `portfolio.json`;
- XO Arena gallery paths and dimensions live in `galleryAssets` in its localized page;
- XO Arena video/poster paths live in `XoArenaVideo`;
- gallery descriptions, media labels, and alt text live in `xo-arena.json`.

To replace media for one locale:

1. Obtain an authentic localized capture/document and optimize it under `public/assets/`; keep the original locale asset intact.
2. Record its provenance in `ASSET_LICENSES.md`.
3. For a rﾃｩsumﾃｩ or project card, update only that locale's `resumeUrl`, `resumePreview`, or project `image`, plus its localized alt text.
4. For XO Arena gallery/video media, make the source mapping locale-aware in the page/video component, then select the reviewed locale asset. Do not put binary data in message catalogs.
5. Preserve explicit image dimensions and the MP4/WebM fallback pair; update the poster from the same localized recording.
6. Run asset-budget tests and visually verify captions, cropping, loading masks, mobile layout, and RTL behavior.

## Adding a locale

1. Add the locale to `routing.locales` and set its direction in `localeDirections`. With `localePrefix: "as-needed"`, only `defaultLocale` is unprefixed; change the default only as a deliberate migration.
2. Add `src/messages/<locale>/common.json`, `portfolio.json`, `xo-arena.json`, and `<locale>.ts` by copying the English schema before translating values.
3. Add the locale's display name to `LocaleSwitcher.locales` in every `common.json`, and add its compact mark to `localeMarks` in the switcher.
4. Add its Open Graph locale code to both metadata maps noted above.
5. If it uses a new script, load an appropriate `next/font` face in the locale layout, add the matching `html[lang]` typography rules, and embed a compatible font in the Open Graph generator.
6. For another RTL locale, replace or extend the current `locale === "fa"` direction checks in interactive components so they derive direction from the shared routing configuration.
7. Review portfolio facts, dates, number formatting, plural/interpolation behavior, bidi isolation, alt text, validation, API/mailto copy, and embedded-language media.
8. Confirm the home page, case study, error/loading/404 states, manifest, canonical URL, language alternates, sitemap entries, locale switcher, and contact flow.
9. Run `npm run format:check`, `npm run lint`, `npm run typecheck`, `npm run test`, `npm run build`, and `npm run test:e2e`. Test desktop and mobile, long-copy overflow, keyboard direction, horizontal drag/scroll, reduced motion, and accessibility in every locale.

Before merging, compare the new catalogs recursively with English for key/type parity, array counts/order/IDs, and ICU placeholders. A translation is incomplete if it silently falls back to English UI copy or changes the content schema.
